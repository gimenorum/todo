# TODO PWA（ローカルファースト ＋ Bring Your Own Storage）実装依頼

## 目的
オフライン主体で動くクロスプラットフォームの TODO アプリを PWA として実装する。自前サーバーは持たず、データはユーザーが選んだ任意のストレージに同期する。PC のブラウザからも同じ URL で使える。主目的は TODO 管理であり、同期はあくまで手段。

## 作業の進め方（重要）
- 実装は **UI を含めてすべて生成側（あなた）が行う**。依頼者はレビューと保守、UI・挙動へのフィードバックを担当。
- コードは「人が読んでレビュー・保守しやすい」ことを最優先（明確なモジュール境界、見通しの良い構造、非自明なロジック＝特にマージ/DAG/同期へのコメント）。
- **フェーズ順に実装**し、各フェーズは「実装フェーズ」で指定した**セクションのみ**を対象とする。特に同期エンジンとそのテストを先に完成させ、正しさをレビューできる状態にしてから先へ進む。

### Git ブランチ戦略（main / develop / feature / hotfix・簡易 Git Flow）
- **main**：リリース済み。GitHub Pages は main から公開。常にデプロイ可能。
- **develop**：開発中フェーズの統合先。feature をここに集める。
- **feature/<名前>**：develop から切り、PR で develop へ戻す。Claude Code のクラウドセッションは feature を develop から切って PR。
- **hotfix/<名前>**：main（最新リリース）から切る。リリース中のバグ修正専用（develop の未完成作業を巻き込まずに本番を直すため）。
- main・develop ともブランチ保護（PR 必須・直接 push 禁止・force-push 禁止・必須ステータスチェック green）。merge 方式は「Create a merge commit」のみ許可（Squash/Rebase 無効化＝手動・squash マージを仕組みで禁止）。
- フロー（通常リリース）：feature →(PR)→ develop に積む → フェーズ完成・green → develop →(PR)→ main → バージョンタグ → デプロイ。
- フロー（フェーズ途中の PATCH）：hotfix を main から切る → 修正 → hotfix →(PR)→ main → PATCH タグ（例 v0.2.1）→ デプロイ → 同じ hotfix を →(PR)→ develop に取り込み（develop にも反映して再発防止）。
- **バージョン（小文字 v ＋ SemVer**。MAJOR=互換を壊す節目／1.0.0=最初の安定版、MINOR=後方互換の機能追加、PATCH=後方互換の修正）：Phase 0 → `v0.0.1`、Phase 1 → `v0.1.0`、Phase 2 → `v0.2.0`、Phase 3 → `v0.3.0`、Phase 4 → `v0.4.0`、**Phase 5 → `v1.0.0`（最初の安定版）**、Phase 6 以降（任意）→ `v1.1.0`…。フェーズ途中/後のバグ修正は hotfix で PATCH（`v0.2.1` 等）。
- 巻き戻し（すべて PR 経由の Revert、reset/force-push 不可）：フェーズ（最新）→ main の「develop→main マージコミット」を GitHub「Revert」（dev からも外すなら develop 側にも Revert PR）。単一機能（develop 上・未リリース）→ その「feature→develop マージコミット」を Revert。古いフェーズだけの巻き戻しは後続依存で本質的に困難（前進修正）。
- release ブランチは作らない（リリースは develop→main が兼ねる）。

## 技術スタック
- 言語：**TypeScript**（同期エンジンの不変条件を型で守るため必須）。
- UI：**フレームワークなし（Vanilla TS）**。DOM 更新は自前（下記パターン）。
- ビルド：**Vite**。IndexedDB：`idb`。テスト：**Vitest**。スタイル：素の CSS。
- サービスワーカー：**手書き（依存なし）**。ルーティング：**ハッシュベース**（History API は使わない＝GitHub Pages で 404 にならない）。
- 上記以外の依存は極力増やさない。

## 対応プラットフォームと制約
- 対象：Windows / macOS / Linux（Debian, CentOS, ChromeOS, SteamOS）/ Android / iOS。標準 Web API のみで全エンジン（Chromium / WebKit / Gecko）で動くこと。
- File System Access API は使わない。
- iOS 固有：インストールは Safari の「ホーム画面に追加」想定。ローカルストレージが消え得るが、正本はリモートにあるため再同期で復旧できる設計に。

## デプロイ / ホスティング
- ホスト：**GitHub Pages（public リポジトリ）**。Vite でビルドし **GitHub Actions** で `dist/` をデプロイ（main のバージョンタグ基準）。
- 独自ドメイン使用（取得済み・DNS は依頼者側で GitHub Pages に向け済み）。**本番オリジンはビルド・文書に固定しない（オリジン非依存）。**
- **オリジン非依存で実装する：マニフェストの `scope`/`start_url` は相対、OAuth のリダイレクト URI は `window.location.origin` 基準、CSP は保存先（Dropbox/Google）ドメイン＋`'self'`。** OAuth リダイレクト URI の各プロバイダ登録は Phase 2/3 に依頼者側で実施。
- CSP は `<meta http-equiv>` で指定（GitHub Pages はヘッダ設定不可のため）。

## 画面構成とナビゲーション
- トップレベルは2つ：**タスク**（TODO 一覧・主画面）と **設定**。「同期」専用タブは作らない。
- レスポンシブナビ（ビューポート幅で切替、ブレークポイント ~768px、CSS メディアクエリ＋必要箇所のみ `matchMedia`）：PC＝折り畳み可能なサイドバー（折り畳み状態は端末ごとの UI 設定としてローカル保存）、モバイル＝下部フッターのタブ。
- 個別画面（編集・競合解決）はハッシュルートで開く（例 `#/todo/:id`）。
- **ナビのバッジ（要対応時のみ／正常時は出さない）**：未解決の競合（件数）→ タスクタブ。要再接続 / 同期エラー → 設定タブ。

## 設定画面
- セクション：クラウド連携（接続/再接続）、同期設定（自動同期の間隔・「手動のみ」）、データ（エクスポート3種・インポート）、（後で）言語。
- 未連携時はここから保存先に接続できる導線を常設。

## UI / DOM 更新の方針（バニラを保守可能に保つ規約）
- 状態の真実（materialize 済み TODO リスト）を1か所に持ち、`setState → render` の一本道にする。ユーザー操作・同期マージ・別タブ更新（BroadcastChannel）を、すべてこの単一経路に流す。軽量な observable / pub-sub を自前で用意してよい。
- リスト描画は全消し再生成でなく、**項目 id をキーに DOM ノードを Map で管理して差分更新**（古いノード残留、編集中フォーカス・スクロール喪失、ちらつきを防ぐ）。
- DOM 生成は文字列連結でなく `<template>` クローン。ユーザー由来テキストは `textContent` で挿入し `innerHTML` は使わない。

## データモデル
- TODO 項目：`{ id: UUID, title: string, done: boolean, dueDate: number|null(時刻含む・未指定はローカル0:00), notifyBeforeMs: number|null(期日の何ms前に通知/null=通知しない / Issue #71), priority: 'none'|'low'|'med'|'high', notes: string, tags: string[], order: string(フラクショナルインデックス), createdAt: number, updatedAt: number, deleted: boolean(tombstone), version: number }`。編集のたびに version をインクリメント。
- `order` は手動並べ替えを後から足すための予約フィールド。v1 は期日/作成順で自動ソートし使わない。
- スナップショット：TODO 項目集合。内容アドレス指定で保存。
- コミットオブジェクト：`{ parents: string[], snapshot: string, timestamp: number, deviceId: string }`。
- オブジェクトはすべて内容の SHA-256 をキーにして保存（不変）。

## 同期エンジン仕様（中核・UI 非依存の純 TS モジュール）
1. 内容アドレス指定：blob は内容ハッシュをキーに保存。作成は衝突しない（同内容=同キー、異内容=異キー）。
2. 先端の導出：コミット集合を一覧し「どのコミットの親でもないコミット」を先端とする。可変 HEAD を置く場合も advisory（キャッシュ）扱いで、常にコミット集合から再計算可能に。
3. 分岐検出とマージ：先端が複数なら fork。DAG をたどり共通祖先（LCA）を求め、両スナップショットを**フィールド単位の 3-way マージ**。
4. マージ規則：別々の項目・片側だけ変わったフィールドは自動マージ。同一項目の同じフィールドが両側で別の値に変わった場合のみ「競合」として検出し、自動解決せず UI に渡す。tombstone と共通祖先が取れない場合は (version, 次点で updatedAt) で決める。
5. 書き込み順序：オブジェクトを先に書き、advisory HEAD 更新は最後。途中失敗は孤立オブジェクトが残るだけ（無害・GC 可能）。
6. 整合性：読み込み時に再ハッシュしてキー一致を検証。
7. CAS は任意の最適化：条件付き書き込みが使える保存先では advisory HEAD の高速パスに使ってよいが、正しさは依存しない（list/get/put/delete で成立）。
8. GC：履歴は残す。オフライン端末が親として参照し得るコミットは消さない。サイズが問題のときだけ古い履歴を base に squash し、孤立 blob のみ GC。
9. 出力に「自動マージ結果」と「未解決の競合リスト（項目 id・フィールド単位の左右の値・基準値）」を含める。

## 競合の表示と解決（WinMerge ライク）
- 競合は競合した TODO ごとに個別表示（全体バナーは作らない）。各競合 TODO の同期ステータスを「**同期できませんでした**」とし、「**同期不具合を解決する**」ボタンを出す。複数競合すれば複数出てよい。
- ボタンでその項目の**マージ画面**へ遷移（タスク一覧から飛ぶ）。左＝この端末／右＝相手の2ペインをフィールド単位（タイトル/期日/優先度/メモ/タグ）で並べ差分を強調、各フィールドを選択（または直接編集）、**メモはテキスト差分表示**、下に**マージ結果プレビュー**、確定でマージコミット。
- 「同期できませんでした」＋解決ボタンは競合専用。一時的なネット/認証エラーは全体ステータス側（オフライン/エラー/要再接続）に出す。

## ステータス表示
- **未連携（保存先未設定）：同期系の表示は一切出さない**。ただのローカル TODO アプリとして振る舞う。連携導線は設定に置き、必要なら初回だけ控えめで消せる案内のみ。
- **連携済み**：
  - 全体ステータス（全画面のヘッダ隅に小さく。普段は静かで必要時のみ主張）：同期中 / オフライン / 最終同期 HH:MM / エラー / 要再接続。**ちらつき対策**：「同期中」は同期が約 400ms を超えたときだけ表示し、表示後は最低約 500ms 維持。短時間で終わる同期では「同期中」を出さず最終同期時刻のみ静かに更新。
  - TODO 個別ステータス（一覧の各項目）：同期済み / 未同期（保存先はあるが未 push）/ 同期できませんでした（＋解決ボタン）。個別に一時的な「同期中」は出さない。
  - 編集画面：開いている項目の同期ステータスを画面上部に表示。
  - ナビのバッジ：要対応のみ（競合件数＝タスクタブ、再接続/エラー＝設定タブ）。

## ローカルストア
- IndexedDB に materialize 済み TODO リストとオブジェクトストア（コミット・blob）のローカル複製を保持。アプリは常にローカルを読み書きし、同期処理がローカルのオブジェクトをリモートと双方向にやり取りする。
- 同一端末の複数タブ間は `BroadcastChannel` で即時同期。

## 同期の設定・タイミング
- **設定は端末ごと（同期しない）**。PC（固定回線）とモバイル（従量・上限）で別にしたいためローカル保存し、TODO データには載せない。
- 設定項目：自動同期の間隔（「手動のみ」も選べる）。**回線種別の判定は行わない。**
- 同期タイミング：①随時（編集後にデバウンスして push）②設定間隔で pull ③画面が隠れた/閉じたとき（`visibilitychange→hidden`。完全なハードクローズだと多段同期が途中で切れ得るため "隠れた時" に流す＝タブ切替や離脱はカバー）④オンライン復帰時 ⑤手動ボタン。
- **オフライン→オンライン**：`online` イベントで検知し、復帰時にバナー（例「オンラインに復帰しました。同期します」）を出して同期を走らせる。`navigator.onLine` は回線が上がったことしか示さないので合図として出し、実際の同期で確かめる。iOS は背面で動かないため復帰後アプリに戻った瞬間に出る想定。

## ストレージアダプタ
- 共通インターフェース：`list(prefix)` / `get(key)` / `put(key, bytes)`（べき等）/ `delete(key)`。
- 実装順：①InMemory（テスト用）②Dropbox（OAuth は PKCE/public client、スコープは「アプリ専用フォルダ」）③Google Drive（appDataFolder＝最小権限、追記専用＋一覧で先端導出、内容ハッシュ名＋存在確認で重複回避、一覧の遅延整合を許容）。S3/WebDAV は同 IF で後から追加可能に（今回は実装不要）。

## エクスポート / インポート（必須・設定画面の「データ」）
- **エクスポートは3種**：①タスク ②設定 ③タスク＋設定。形式は **JSON を正本**（全種・無損失・再取り込み可）、タスクは加えて Markdown チェックリスト（`- [ ]`/`- [x]`）と CSV、設定は JSON。受け渡しは Blob ダウンロード＋`navigator.share`。File System Access API は使わない。
- **インポート**：`<input type="file">` で読み込み。タスク＝マージエンジンで取り込む。設定＝このデバイスに適用（上書き・確認あり）。

## セキュリティ（v1.0 では後回し）
- クライアント側暗号化・本格的なハードニングは Phase 6（post-1.0）。
- ただしコストゼロの基本は守る：`innerHTML` 不使用（`textContent`）、CSP の `<meta>`、HTTPS、OAuth は最小権限スコープ（Dropbox アプリ専用フォルダ / Drive appDataFolder）＋ PKCE/public client。トークンは IndexedDB 保持（防御境界ではない前提）。

## 受け入れ基準（不変条件）
- すべての読み書きがオフラインで動作する。
- 別々の項目／片側だけの変更は自動で両立する。同一項目の同一フィールドが食い違ったときだけ競合として該当 TODO に表示され、ユーザーが解決するまで黙って失われない。
- 同期の正しさが原子的 CAS の有無に依存しない（list/get/put/delete で成立）。
- オブジェクトは読み込み時に再ハッシュ検証される。
- DOM はリスト変更時に id キーで差分更新され、編集中フォーカス・スクロールが保たれる。
- 未連携時は同期系 UI を一切出さない。連携済み時のみステータス・バッジが現れる。
- 同期設定は端末ごとで同期データに載らない。設定は単独でエクスポート/インポート（適用）できる。
- ナビは ~768px でサイドバー⇄ボトムタブに切り替わる。
- iOS で Background Sync に依存せず、前面復帰/起動時に同期。ローカルが消えてもリモートから復旧できる。
- タスクの JSON エクスポート→インポートが無損失で往復し、インポートはマージで取り込む。

## テストすべき並行シナリオ（InMemory アダプタ上で決定的に）
1. 2端末がオフラインで別々の項目を編集 → 両方残る。
2. 2端末が同じ親から同時にコミット（fork）→ 共通祖先からの 3-way マージで一貫した結果。
3. 同一項目の異なるフィールドを両端末で編集 → 自動マージされ、競合にならない。
4. 同一項目の同じフィールドを両端末で別の値に編集 → 競合として検出され、該当項目に「同期できませんでした」が立つ（自動解決しない）。
5. 片方が編集・他方が削除（edit vs delete）→ 競合として検出し、該当項目に「編集版を残す／削除を適用」の二択を出す（共通祖先を使う）。
6. 古い状態からの上書き → 先端導出により検出され、マージしてから反映、変更が消えない。
