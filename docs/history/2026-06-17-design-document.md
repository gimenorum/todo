2026-06-17

## 依頼内容
- 要件定義書（`docs/requirements.md`）から「設計書」を起こすこと。
- 設計フェーズのため、一時的にブランチを作成して作業し、main への取り込み（直コミット）は後から行う方針。
- 確認の結果、以下を決定：
  - 構成は**章分割**（`docs/design/` ＋ 目次 README ＋ 00–18 章）。
  - 粒度は**全章を通しで作成**（中核は厚く、その他は確定方針＋構造まで）。
  - 一時ブランチには**コミットまで**（push はしない）。

## 対応概要
- `main` 起点で一時ブランチ `design/todo-pwa` を作成。
- `docs/design/` に設計書一式（目次 README ＋ 19 章）を新規作成。
  - 中核（同期エンジン §04・データモデル型 §03・アーキ §01-02・テスト §16）は擬似コード／TS 型骨子／決定表まで厚く記述。
  - UI／PWA／CI 等（§07–15）は確定方針＋構造まで記述。
  - 各章冒頭に「要件トレース（L番号）」を置き、要件の引き写しを避けて決定・構造・型・擬似コードに集中。
  - 受け入れ基準（L145–155）を各章の「不変条件」節に紐付け。
  - 要件に明記の無い設計判断（マージ対象フィールドの取捨・edit vs delete・tags マージ・LCA tie-break 等）は根拠付きで明記し、`18-open-questions.md` に要決定リストとして集約。
- 本履歴を設計書と同一コミットに含めてコミット（push はしない）。

## 決定事項
- 設計書の保存先は `docs/design/`、構成は章分割（目次 README ＋ `00`〜`18`）。
- 設計フェーズの作業は一時ブランチ `design/todo-pwa` で行い、**コミットまで・push なし**。main への取り込みは後から指示を待つ。
- 中核章を厚く、その他は方針＋構造まで、という粒度方針。
- 中核の設計上の既定（覆る可能性ありだが当面の前提）：
  - マージ対象 `TodoField` は 7 フィールドに限定（createdAt/order/updatedAt/version を除外）。
  - edit vs delete は `deleted` のフィールド競合として扱う（推奨）。
  - tags は集合 3-way（推奨）。LCA tie-break は `(timestamp, hash)` の全順序。
