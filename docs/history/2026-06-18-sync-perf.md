## 対応概要
- ブランチ `hotfix/sync-perf` を `origin/main`（v0.4.2）から作成。
- 原因: コア同期エンジン `pushReachable` は到達可能な全 object を毎回 `put` し直すが、体感の遅さの主因は各アダプタの「1 操作あたりのネットワーク往復」が過大なこと。
  - Google Drive: `get`=findId+download の 2 リクエスト。`put`=毎回 findId で存在確認。`list` は id をキャッシュしない。
  - Dropbox: `put`=`mode:overwrite` で毎回フル本文を再アップロード（不変な `objects/*` も毎同期上げ直し）。
- 修正方針: `StorageAdapter` IF もコア同期エンジンも不変のまま、各アダプタ内部のキャッシュだけで「差分のみ」に縮める。
  - **Google Drive — name→id キャッシュ**（`idCache`）。`list()` が全ファイルの name→id をウォーム、`findId()` はキャッシュ命中で即返し、ミス時のみネットワーク検索。`createMultipart()` が作成 id を、`delete()` が除去を反映。⇒ `get` 2→1、既存 `objects/` の `put` は 0 リクエスト。
  - **Dropbox — 既知 object 集合**（`knownObjects`、事前 list なし）。既知の `objects/*` は `put` でアップロード省略、成功後に追加。`heads/*` は常にアップロード。`get` 非 null で追加、`delete` で除去。
  - 不変条件: `objects/*` は内容アドレス指定＝不変、エンジンは削除しないため偽陽性は起きない。ミス/未知は必ずネットワーク／アップロードへフォールバックし正しさを保持。
- テスト追加（呼び出し回数の回帰防止）:
  - Drive: `list('heads/')` ウォーム後、既知 object の `get`・既存 `put` が `q=name`（findId 検索）を出さない。
  - Dropbox: 同一 `objects/*` を 2 回 `put` して 2 回目に `files/upload` が出ない／`heads/*` は毎回出る。
- 既存の契約テスト・固有挙動テストは観測結果（store 状態）不変でそのまま緑。
- 設計更新: `docs/design/05-storage-adapter.md` に §5.8「アダプタ内キャッシュで差分 push にする」を追記。
- `package.json` 0.4.2 → 0.4.3。
- ローカルで `typecheck` / `lint` / `test`（142 件）/ `build` をすべて green を確認。

## 決定事項
- コア同期エンジン（収束ロジック）には手を入れず、アダプタ内部キャッシュだけで往復を削る（収束テストの回帰リスクを負わない）。
- Dropbox のウォームは事前 list を使わない（既存の content 401 リトライ等の auth 経路に干渉しないため）。
- #26 の検証完了前でも #27 を先行リリースしてよい（ユーザー判断 2026-06-18）。
