# TODO PWA 設計書

要件定義（[`../requirements.md`](../requirements.md)）を **実装可能な技術設計** へ落とした設計書です。
要件が「何を・なぜ（WHAT / WHY）」を述べるのに対し、本設計書は「どう作るか・どんな形か（HOW / SHAPE）」を述べます。

- 言語: 日本語（[`../../CLAUDE.md`](../../CLAUDE.md) 規約）
- 中核: **UI 非依存の純 TS 同期エンジン**（内容アドレス指定 DAG／先端導出／LCA／フィールド単位 3-way マージ／競合検出／CAS 非依存／再ハッシュ検証）
- 進め方: 要件「作業の進め方」のとおり「同期エンジンとテストを先に完成・レビュー可能に」。実装は Phase 0〜6 で段階的。

## 章のライフサイクル運用

グリーンフィールドのため、章は段階的に肉付けする。各章の見出し直下に **状態** と **実装フェーズ** を明記する。

- **状態**: `ドラフト`（初稿・方向性確定）→ `確定`（当該フェーズ着手前にレビュー済み）→ `実装済`（コードと一致を確認）
- フェーズ着手時に該当章を「ドラフト → 確定」に引き上げ、実装完了で「実装済」にする。
- 複数フェーズに跨る章は中間状態として `一部実装済（Phase N）` を用いる（当該フェーズ分の実装が完了し、コードと一致を確認した状態）。

## 設計書の書き方（約束）

1. **要件を引き写さない**。各章冒頭に `要件トレース:` でセクション名を置き、本文は決定・構造・型・擬似コードに集中する（二重メンテ＝陳腐化を防ぐ）。requirements は行番号でなく**見出し（セクション名）で参照**する。
2. 要件に明記の無い**設計判断には根拠を 1 行**添える。
3. 擬似コードは「不変条件・分岐網羅を明示、データ構造の些末は省略」。**実 TS コードは書かない**（設計書はコードの先回りをしない）。
4. 受け入れ基準（要件の同名セクション）に対応する**不変条件**を各章で明示的に紐付ける。
