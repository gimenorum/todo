# 11. 同期トリガとスケジューリング

> 要件トレース: requirements.md「同期の設定・タイミング」「受け入れ基準」
> 状態: ドラフト ／ 実装フェーズ: 2

`services/SyncScheduler.ts` が同期の起動を司る。実際の 1 回の同期は [04 §4.6](./04-sync-engine.md) の `syncOnce`。

## 11.1 設定は端末ごと（同期しない）

- 自動同期の間隔（「手動のみ」も選べる）を `DeviceSettings`（[03](./03-data-model.md)）にローカル保存。**TODO データには載せない**（要件「同期の設定・タイミング」・受け入れ基準）。
- PC（固定回線）とモバイル（従量・上限）で別にしたいため端末ごと。
- **回線種別の判定は行わない**（要件「同期の設定・タイミング」）。

## 11.2 5 つのトリガ

| # | トリガ | 動作 | 備考 |
|---|---|---|---|
| ① | 編集後 | デバウンスして push | **既定デバウンス = 2 秒**（確定 / [18](./18-open-questions.md) #9）。定数化 |
| ② | 設定間隔 | 定期 pull | **既定間隔 = 5 分**（確定 / #9）。`autoSyncMode='manual'` のとき無効（要件「同期の設定・タイミング」） |
| ③ | `visibilitychange → hidden` | flush 同期 | ハードクローズだと多段同期が切れ得るため「隠れた時」に流す（タブ切替・離脱をカバー / 要件「同期の設定・タイミング」） |
| ④ | `online` 復帰 | バナー＋同期 | 下記 §11.3 |
| ⑤ | 手動ボタン | 即同期 | ヘッダ／設定 |

## 11.3 オフライン→オンライン復帰

- `online` イベントで検知し、復帰時にバナー（例「オンラインに復帰しました。同期します」）を出して同期を走らせる（要件「同期の設定・タイミング」）。
- `navigator.onLine` は回線が上がったことしか示さないので**合図**として扱い、実際の同期で確かめる。
- iOS は背面で動かないため、**復帰後にアプリへ戻った瞬間**に出る想定（[12](./12-pwa-sw-csp.md)）。これがローカル消失からの復旧契機にもなる（[06 §6.4](./06-local-store.md)・受け入れ基準）。

## 11.4 多重実行の防止

- 同期中フラグを持ち、走行中の追加トリガは「末尾で 1 回だけ再実行」に畳む（最新状態で締める）。
- push（編集後）と pull（間隔）が重なっても `syncOnce` は冪等的に先端を再導出するため安全（[04](./04-sync-engine.md)）。

```
schedule(trigger):
  if syncing: pendingRerun = true; return
  syncing = true
  try: await syncOnce(adapter, local)
  finally:
    syncing = false
    if pendingRerun: pendingRerun = false; schedule('rerun')
```

## 11.5 関連する不変条件

- 同期設定は端末ごとで同期データに載らない（受け入れ基準）。
- オン/オフライン処理があり、復帰時に同期が走る（受け入れ基準 / 要件「同期の設定・タイミング」）。
- iOS は Background Sync に依存せず、前面復帰/起動時に同期（受け入れ基準）。
- ローカル読み書きは同期の成否に関わらず常に動く（受け入れ基準・[06](./06-local-store.md)）。
